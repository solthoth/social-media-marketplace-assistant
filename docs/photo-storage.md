# Photo Storage

Photo storage starts with local filesystem storage for development and keeps the backend contract ready for future object or block storage.

## Goals

- Keep item photos in an application-owned storage location.
- Store photo metadata in SQLite and photo bytes outside the database.
- Avoid exposing local filesystem paths to the frontend.
- Preserve an adapter boundary so future Azure, AWS, or S3-compatible storage can replace local storage without changing handlers or frontend contracts.
- Plan for multiple generated sizes so the UI can load thumbnails without downloading full originals.

## Local Storage

Local storage is configured through `PHOTO_STORAGE_PATH`.

Default:

```sh
PHOTO_STORAGE_PATH=data/photos
```

The default path is ignored by git. Docker Compose stores photos under `/data/photos` in the existing API data volume.

The local adapter should create the storage directory on startup or first write. Stored file names should be generated by the application, not derived from user-provided file names.

Recommended local key shape:

```text
items/{item_id}/photos/{photo_id}/{variant}.{extension}
```

Examples:

```text
items/123/photos/abc/original.jpg
items/123/photos/abc/thumbnail.jpg
items/123/photos/abc/medium.jpg
```

## Metadata

The database remains the source of truth for photo metadata. The existing `item_photos` table should track:

- `id`
- `item_id`
- `storage_id`
- `filename`
- `mime_type`
- `sort_order`
- `is_primary`
- `created_at`

`storage_id` should be opaque to callers. It should identify the storage object or prefix needed by the storage adapter, not a public URL or raw filesystem path.

## Variants

Photo upload should keep enough information to support multiple sizes:

- `original`: retained source image after validation.
- `medium`: display-sized image for item detail and listing preparation views.
- `thumbnail`: small image for inventory cards and selection lists.

Image normalization and compression should be planned but does not need to be implemented in the first upload API. The design should leave room for a processor that can generate variants asynchronously or inline after upload.

Recommended future normalization rules:

- Strip unsafe metadata.
- Correct orientation.
- Convert supported images to a consistent web-friendly format.
- Bound maximum dimensions and file size.
- Generate deterministic variants from the original.

## Backend Interface

Handlers should depend on an application service, and services should depend on a storage interface rather than a concrete filesystem implementation.

Suggested interface shape:

```go
type PhotoStorage interface {
    Save(ctx context.Context, object PhotoObject, content io.Reader) (StoredPhotoObject, error)
    Open(ctx context.Context, storageID string, variant PhotoVariant) (io.ReadCloser, PhotoObjectInfo, error)
    Delete(ctx context.Context, storageID string) error
}
```

Suggested value objects:

- `PhotoVariant`: `original`, `medium`, or `thumbnail`.
- `PhotoObject`: item ID, photo ID, variant, file extension, and content type.
- `StoredPhotoObject`: opaque storage ID and size metadata.
- `PhotoObjectInfo`: content type, size, and optional cache metadata.

The first implementation should live behind a local filesystem adapter, for example `internal/storage/localphotos`. A future cloud adapter can implement the same interface with Azure Blob Storage, AWS S3, or another object/block storage provider.

## API Content Delivery

The backend should serve photo content through API routes for the MVP.

Recommended route shape:

```text
GET /items/{itemId}/photos/{photoId}/content?variant=thumbnail
```

Reasons:

- The frontend does not need to know whether storage is local or remote.
- Local filesystem paths are never exposed.
- Authorization can be enforced consistently once auth is added.
- The API can later return redirects or signed URLs for cloud storage without changing the frontend resource model.

The upload/list/delete/reorder API can return metadata plus content URLs generated by the API. Those URLs should be treated as API resources, not storage implementation details.

## Future Cloud Storage

Cloud storage should keep the same logical model:

- The database stores metadata and opaque storage IDs.
- The storage adapter owns provider-specific bucket/container names and object keys.
- The API controls access and can either stream content or issue short-lived signed URLs.

Provider-specific configuration should be added only when implementing the cloud adapter. Expected future settings may include:

- Storage provider.
- Bucket or container name.
- Region or account endpoint.
- Credential source.
- Optional CDN or public asset base URL.
